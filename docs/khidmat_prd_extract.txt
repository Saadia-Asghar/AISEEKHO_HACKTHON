Challenge 2: AI Service Orchestrator for Informal Economy
1. Executive Summary
KhidmatAI is an agentic AI platform that automates the entire lifecycle of informal service requests — plumbers, electricians, AC technicians, tutors, beauticians — in Pakistan’s largely undigitized service economy. A user types or speaks a request in Urdu, Roman Urdu, or English; KhidmatAI’s multi-agent pipeline (orchestrated entirely via Google Antigravity) understands intent, discovers nearby providers, ranks them, simulates a booking confirmation, and schedules follow-up reminders — all without any human dispatcher.
2. Problem Statement
2.1  The Informal Economy Gap

2.2  Why Existing Solutions Fail
Listing apps (e.g., OLX, Rozgaar.pk) require manual browsing — no automation or booking
WhatsApp bots are single-purpose and lack reasoning across steps
Enterprise tools are unaffordable and irrelevant to informal workers
No solution supports Urdu/Roman Urdu as primary input language with agentic reasoning

4. System Architecture
4.1  High-Level Overview
KhidmatAI is built as a multi-agent system where Google Antigravity acts as the orchestration layer, routing tasks between specialized sub-agents, managing state, and executing tool calls. External LLMs may be used for NLU and generation tasks, but all planning, routing, and decision-making flows through Antigravity.

4.3  Antigravity Integration Details
Orchestration: All agent-to-agent calls registered as Antigravity workflow nodes
Tool Registration: Maps API, Places API, mock booking DB registered as Antigravity tools
Reasoning Trace: Antigravity’s built-in logging captures planning steps → decision → action for each node
State Management: Booking state (pending → confirmed → reminder_sent) managed via Antigravity session context
Multi-step Reasoning: Antigravity controls the planning loop — agents re-query if initial results are insufficient

5. Functional Requirements
5.1  FR-1: Intent Understanding
The system MUST accept free-form text in Urdu, Roman Urdu, or English and extract a structured intent object.
5.2  FR-2: Provider Discovery
Query Google Maps Places API (or mock dataset) with service_type + location
Filter results within configurable radius (default: 5 km)
Return minimum 3 candidate providers with: name, distance, rating, availability flag
Gracefully fall back to mock JSON dataset if API unavailable
5.3  FR-3: Matching & Ranking
Score each provider on a weighted formula:
Distance score: 40% weight (inverse of km)
Rating score: 35% weight (normalized 1–5)
Availability: 25% weight (binary: available now vs. scheduled)
Antigravity reasoning trace MUST show scoring calculation for top 3 candidates
Final recommendation presented with plain-language explanation
5.4  FR-4: Action Simulation (CRITICAL per challenge brief)
System MUST simulate a complete booking — not just recommend
Booking simulation writes a record to mock DB / Google Sheet with fields:
booking_id, customer_name, provider_id, service, slot_datetime, status=CONFIRMED
System generates a booking receipt (text or PDF) with all booking details
Provider assignment confirmed and reflected in state
Full state transition logged: PENDING → CONFIRMED → REMINDER_SCHEDULED
5.5  FR-5: Follow-Up Automation
Schedule a reminder notification 1 hour before appointment (simulated via FCM or WhatsApp mock)
After appointment slot passes, send a “service completed?” confirmation prompt
Log all follow-up events in Antigravity trace with timestamps
5.6  FR-6: Agentic Workflow Trace (MANDATORY)
This is a non-negotiable deliverable. The system MUST produce a human-readable trace showing:
Each agent activated (Intent → Discovery → Ranking → Booking → Follow-Up → Trace)
Tools called at each step with input/output
Reasoning at each decision point (why provider X was selected)
Timestamps for each step
Final outcome summary

10. Implementation Build Plan
Recommended build sequence for the Antigravity team, ordered to ensure the most critical hackathon requirements are working first.
Phase 1 — Antigravity Core (Build First)
Set up Antigravity project; register all 6 agents as workflow nodes
Register Google Maps + Places API as Antigravity tools
Implement Intent Agent with Gemini LLM call; test Urdu/Roman Urdu parsing
Implement Discovery Agent with mock dataset fallback
Phase 2 — Booking Simulation (Critical Requirement)
Implement Ranking Agent with weighted scoring; output reasoning chain
Implement Booking Agent; connect to Firebase/Sheets mock; generate receipt
Implement Follow-Up Agent; log reminder schedule to Antigravity trace
Implement Trace Agent; compile and format full workflow log
Phase 3 — UI & Demo Polish
Build Flutter mobile app with conversational input and booking status screen
Add Antigravity trace viewer (collapsible step-by-step)
Record 3–5 min demo video with primary scenario + 1 edge case
Write README with architecture diagram and Antigravity integration notes
11. Constraints & Guidelines
Antigravity MUST be central to system logic — not a thin wrapper around another orchestrator
External LLMs (Gemini, etc.) are permitted but must be invoked as Antigravity tools
No real personal/sensitive data in the demo
Mobile app is required; web app is a bonus
Mock data is acceptable and encouraged when real APIs are rate-limited during demo
